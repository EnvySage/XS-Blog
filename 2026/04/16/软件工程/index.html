软件工程

概要设计的内容可以包含系统构架、模块划分、系统接口、数据设计4个主要方面的内容,不包括模块内算法设计。

需求分析阶段需完成以下要求：
►确定软件系统的功能需求和非功能需求；
►分析软件系统的数据要求；
►导出系统的逻辑模型；
►修正项目开发计划；
►如有必要，可以开发一个原型系统。

  1. 客户特性是指与客户的素质以及开发者和客户定期通信的能力相关的风险；
  2. 过程定义是指与软件过程被定义的程度以及它们被开发组织所遵守的程序相关的风险；
  3. 开发环境是指与用以构建产品的工具的可用性及质量相关的风险；
  4. 构建技术是指与待开发软件的复杂性及系统所包含技术的“新奇性”相关的风险。

软件配置管理是一组管理整个软件生存期各阶段中变更的活动，主要包括变更标识、变更控制和版本控制。

软件详细设计阶段的主要任务包括：对模块内的数据结构进行设计；对数据库进行物理设计；对每个模块进行详细的算法设计；代码设计、输入/输出设计、用户界面设计等其他设计。

CMM的5个成熟度等级分别为：

  1. 初始级。初始级是未加定义的随意过程，软件过程的特点是无秩序的，有时甚至是混乱的。软件过程定义几乎处于无章法和步骤可循的状态，软件产品所取得的成功往往依赖于极个别人的努力和机遇。
  2. 可重复级。可重复级是规则化和纪律化的过程，软件过程已建立了基本的项目管理过程，可用于对成本、进度和功能特性进行跟踪。对类似的应用项目，有章可循并能重复以往所取得的成功。
  3. 已定义级。已定义级是标准的和一致的过程，用于管理的和工程的软件过程均已文档化、标准化，并形成了整个软件组织的标准软件过程。全部项目均采用与实际情况相吻合的、适当修改后的标准软件过程来进行操作。
  4. 已管理级。已管理级是可预测的过程，软件过程和产品质量有详细的度量标准。软件过程和产品质量得到了定量的认识和控制。
  5. 优化级。优化级是持续改进的过程，通过对来自过程、新概念和新技术等方面的各种有用信息的定量分析，能够不断地、持续性地对过程进行改进。

CMMI模型分为0-6

  1. 能力等级0指未执行过程，表明过程域的一个或多个特定目标没有被满足；
  2. 能力等级1指过程通过转化可识别的输入工作产品，产生可识别的输出工作产品，关注于过程域的特定目标的完成；
  3. 能力等级2指过程作为已管理的过程制度化，针对单个过程实例的能力；
  4. 能力等级3指过程作为已定义的过程制度化，关注过程的组织级标准化和部署；
  5. 能力等级4指过程作为定量管理的过程制度化；
  6. 能力等级5指过程作为优化的过程制度化，表明过程得到很好地执行且持续得到改进。

  • 统一过程（UP）的基本特征是“用力驱动，以架构为中心的个受控的迭代式增量开发”。一个UP可分为若干个周期，每个周期的开发过程被分为4个阶段，每个阶段可进行若干次迭代。

  • （1）先启阶段（Inception）：该阶段的主要意图是建立项目的范围和版本，确定业务实现的可能性和项目目标的稳定性。提交结果包括原始的项目需求的业务用例。

  • （2）精化阶段（Elaboration）：该阶段的主要意图是对问题域进行分析，建立系统的需求和架构。确定技术实现的可行性和系统架构的稳定性。提交结果包括系统架构及其相关文档、领域模型、修改后的业务用例和整个项目的开发计划。

  • （3）构建阶段（Construction）：主要意图是增量式地开发一个可以交付用户的软件产品。

  • （4）提交阶段（Transition）：主要意图是将软件产品提交用户。

  • RUP中每个阶段的结束时所产生的里程碑。

  • 初启阶段结束时的里程碑是生命周期目标；

  • 精化阶段结束时的里程碑是生命周期架构；

  • 构建阶段结束时的里程碑是最初运作能力；

  • 移交阶段的里程碑是产品发布。

网络工程

模块内聚

1. 偶然内聚

  • 定义：模块内各组成部分之间没有任何关联，仅因为偶然的原因被放在同一个模块中。这种内聚程度最低。
  • 举例：一个模块中既包含了计算工资的代码，又包含了打印报表的代码，而这两部分功能完全无关，只是因为开发时被放在了同一个文件中。

2. 逻辑内聚

  • 定义：模块内各组成部分虽然执行相似的功能，但这些功能属于同一逻辑类别，具体功能由输入参数决定。
  • 举例：一个模块用于处理各种图形的绘制，如绘制圆形、正方形、三角形等。虽然这些功能都属于图形绘制，但具体绘制哪种图形取决于输入参数。

3. 时间内聚

  • 定义：模块内各组成部分必须在同一时间内执行，但这些组成部分之间没有逻辑上的关联。
  • 举例：一个模块用于初始化软件系统，其中包含了初始化数据库、初始化用户界面、初始化网络连接等多个操作。这些操作虽然在启动时一起执行，但彼此之间没有逻辑上的联系。

4. 过程内聚

  • 定义：模块内各组成部分在逻辑上密切相关，且这些组成部分必须按照特定的顺序执行。
  • 举例：一个模块用于处理订单，包括接收订单、验证订单、处理支付、发货等步骤。这些操作在逻辑上密切相关，且必须按照一定的顺序执行。

5. 通信内聚

  • 定义：模块内各组成部分都操作同一数据结构或数据集。
  • 举例：一个模块用于处理学生信息，包括添加学生、删除学生、查询学生等操作。这些操作都基于同一个学生信息数据结构。

6. 顺序内聚

  • 定义：模块内各组成部分不仅操作同一数据结构，而且一个组成部分的输出是另一个组成部分的输入。
  • 举例：一个模块用于处理文本文件，首先读取文件内容，然后对内容进行格式化，最后输出处理后的结果。这些操作不仅基于同一个数据结构（文件内容），而且存在明确的输入输出关系。

7. 功能内聚

  • 定义：模块内所有组成部分都紧密围绕一个单一的功能，共同完成一个明确的任务。这是内聚程度最高的类型。
  • 举例：一个模块用于计算圆的面积和周长，所有操作都围绕“圆”这个单一功能展开，输入圆的半径，输出面积和周长。

模块耦合

耦合级别 描述 特点 优点 缺点
无耦合 模块之间没有任何联系 完全独立 最理想的耦合状态 实际开发中很难完全实现
数据耦合 模块之间通过参数传递简单数据值 依赖性弱 耦合程度低，便于维护和扩展 数据传递过多可能复杂
标记耦合 模块之间通过传递数据结构 依赖性适中 可传递复杂数据结构 数据结构变化影响多个模块
控制耦合 模块之间通过传递控制信号 依赖性强 灵活控制模块行为 模块行为相互影响
外部耦合 模块之间通过外部环境交互 依赖性强 可通过外部环境共享数据 外部环境变化影响模块行为
公共耦合 模块之间通过共享全局变量 依赖性强 实现简单，便于数据共享 全局变量修改影响多个模块
内容耦合 模块之间直接访问或修改对方内部数据 依赖性最强 实现简单，直接访问内部数据 模块修改直接影响其他模块

设计模式

1.创建型模式

1.1 单例模式（Singleton）

  • 意图：确保一个类只有一个实例，并提供一个全局访问点。

1.2 工厂方法模式（Factory Method）

  • 意图：定义一个创建对象的接口，让子类决定实例化哪一个类。

1.3抽象工厂模式（Abstract Factory）

  • 意图：创建相关或依赖对象的家族，而不需明确指定具体类。

1.4 建造者模式（Builder）

  • 意图：逐步构建一个复杂的对象。

1.5 原型模式（Prototype）

  • 意图：通过复制现有实例来创建新实例。

2. 结构型模式

2.1 适配器模式（Adapter）

  • 意图：将一个类的接口转换为另一个接口，以使原本不兼容的类可以协同工作。

2.2 桥接模式（Bridge）

  • 意图：将抽象与实现分离，使它们可以独立变化。

2.3 组合模式（Composite）

  • 意图：将对象组合成树形结构，以表示“部分-整体”的层次结构。

2.4 装饰模式（Decorator）

  • 意图：动态地给一个对象添加额外的功能。

2.5 代理模式（Proxy）

  • 意图：为其他对象提供一种代理以控制对这个对象的访问。

2.6 外观模式（Facade）

  • 意图：提供一个统一的高层接口，用于访问子系统中的一群接口。

2.7 飞享模式（Flyweight）

  • 意图：通过共享已存在的对象来减少创建对象的数量，从而减少内存占用和提高性能。

3.行为型模式（Behavioral Patterns）

3.1 策略模式（Strategy）

  • 意图：定义一系列算法，将每个算法封装起来，并使它们可以互换。

3.2 模板方法模式（Template Method）

  • 意图：定义一个操作中的算法骨架，将一些步骤延迟到子类中实现

3.3 观察者模式（Observer）

  • 意图：定义对象间的一种一对多的依赖关系，以便当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并自动更新

3.4 迭代器模式（Iterator）

  • 意图：提供一种顺序访问一个聚合对象中各个元素的方法，而不暴露其内部的表示

3.5 责任链模式（Chain of Responsibility）

  • 意图：使多个对象都有机会处理请求，从而避免请求的发送者和接收者之间的耦合关系。

3.6 命令模式（Command）

  • 意图：将一个请求封装为一个对象，从而使用户可用不同的请求对客户进行参数化。

3.7 备忘录模式（Memento）

  • 意图：在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态。

3.8 状态模式（State）

  • 意图：允许一个对象在其内部状态改变时改变其行为。

3.9 访问者模式（Visitor）

  • 意图：为一个对象结构中的对象添加新的能力，而无需修改这些对象的类。

3.10 中介者模式（Mediator）

  • 意图：用一个中介对象来封装一系列对象间的交互，使对象不需要显式地相互引用，从而使其耦合松散

3.11 解释器模式（Interpreter）

  • 意图：给定一个语言，定义它的文法表示，并定义一个解释器，这个解释器使用该文法来解释语言中的句子。

总结

创建型模式（Creational Patterns）

  1. 单例模式（Singleton）
    • 意图：确保一个类只有一个实例，并提供一个全局访问点。
    • 适用范围：当需要全局访问某个资源（如配置文件、日志记录器）时，或者当对象的创建成本较高且只需要一个实例时。
  2. 工厂方法模式（Factory Method）
    • 意图：定义一个创建对象的接口，让子类决定实例化哪一个类。
    • 适用范围：当类的实例化过程复杂，或者需要根据不同的条件创建不同的对象时。
  3. 抽象工厂模式（Abstract Factory）
    • 意图：创建相关或依赖对象的家族，而不需明确指定具体类。
    • 适用范围：当需要创建一组相关或依赖的对象，而不需要指定它们的具体类时。
  4. 建造者模式（Builder）
    • 意图：将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。
    • 适用范围：当需要构建一个复杂的对象，并且对象的构建过程需要逐步进行时。
  5. 原型模式（Prototype）
    • 意图：通过复制现有的实例来创建新的实例，而不是通过新建实例。
    • 适用范围：当创建对象的过程比较复杂，或者需要频繁创建相似的对象时。

结构型模式（Structural Patterns）

  1. 适配器模式（Adapter）
    • 意图：将一个类的接口转换成客户希望的另一个接口。
    • 适用范围：当需要使用一个已有的类，但其接口不符合当前需求时。
  2. 桥接模式（Bridge）
    • 意图：将抽象与实现解耦，让它们可以独立变化。
    • 适用范围：当需要将抽象与实现分离，或者需要在运行时动态地选择实现时。
  3. 组合模式（Composite）
    • 意图：将对象组合成树形结构以表示“部分 - 整体”的层次结构。
    • 适用范围：当需要表示对象的部分 - 整体关系时，例如文件系统中的文件和目录。
  4. 装饰器模式（Decorator）
    • 意图：动态地给一个对象添加额外的职责。
    • 适用范围：当需要在运行时动态地给对象添加功能时，而不是通过继承。
  5. 外观模式（Facade）
    • 意图：为子系统中的一组接口提供一个一致的界面。
    • 适用范围：当需要简化复杂的子系统接口，或者需要提供一个统一的高层接口时。
  6. 享元模式（Flyweight）
    • 意图：通过共享来高效地支持大量细粒度的对象。
    • 适用范围：当需要创建大量相似的对象，并且这些对象的创建和存储成本较高时。
  7. 代理模式（Proxy）
    • 意图：为其他对象提供一种代理以控制对这个对象的访问。
    • 适用范围：当需要控制对对象的访问，或者需要在对象的创建和使用过程中添加额外的逻辑时。

行为型模式（Behavioral Patterns）

  1. 责任链模式（Chain of Responsibility）
    • 意图：使多个对象都有机会处理请求，从而避免请求的发送者和接收者之间的耦合关系。
    • 适用范围：当需要处理一系列请求，并且每个请求可能由不同的对象处理时。
  2. 命令模式（Command）
    • 意图：将一个请求封装为一个对象，从而使用户可用不同的请求对客户进行参数化。
    • 适用范围：当需要将请求的发送者和接收者解耦，或者需要支持撤销操作时。
  3. 解释器模式（Interpreter）
    • 意图：给定一个语言，定义它的文法的一种表示，并定义一个解释器，这个解释器使用该表示来解释语言中的句子。
    • 适用范围：当需要解释一种语言或表达式时，例如SQL查询、正则表达式等。
  4. 迭代器模式（Iterator）
    • 意图：提供一种顺序访问一个聚合对象中各个元素的方法，而不暴露其内部的表示。
    • 适用范围：当需要遍历一个聚合对象，但又不想暴露其内部结构时。
  5. 中介者模式（Mediator）
    • 意图：用一个中介对象来封装一系列的对象交互。
    • 适用范围：当多个对象之间存在复杂的交互关系，需要减少它们之间的耦合时。
  6. 备忘录模式（Memento）
    • 意图：在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态。
    • 适用范围：当需要保存和恢复对象的状态时，例如实现撤销操作。
  7. 观察者模式（Observer）
    • 意图：定义对象间的一种一对多的依赖关系，以便当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并自动更新。
    • 适用范围：当需要在对象之间建立动态的依赖关系时，例如事件驱动系统。
  8. 状态模式（State）
    • 意图：允许一个对象在其内部状态改变时改变其行为。
    • 适用范围：当一个对象的行为取决于其状态，并且状态的改变会导致行为的改变时。
  9. 策略模式（Strategy）
    • 意图：定义一系列的算法，把它们一个个封装起来，并使它们可以互换。
    • 适用范围：当需要在运行时选择不同的算法或行为时。
  10. 模板方法模式（Template Method）
    • 意图：定义一个操作中的算法的骨架，而将一些步骤延迟到子类中。
    • 适用范围：当需要定义一个算法的框架，但允许子类在某些步骤上进行定制时。
  11. 访问者模式（Visitor）
    • 意图：表示一个作用于某对象结构中的各元素的操作。
    • 适用范围：当需要对一个对象结构中的对象进行多种不同的操作，而不想改变这些对象的类时。

·面向对象测试

算法层:测试类中定义的每个方法，基本相当于传统软件测试的单元测试。

类层:测试封装在同一个类中的所有方法与属性之间的相互作用。可以认为是面向对象测试中特有的模块测试。

模板层:测试一组协调工作的类之间的相互作用。大体上相当于传统软件测试中的集成测试。

系统层:把各个子系统组装成完整的面向对象软件系统。
